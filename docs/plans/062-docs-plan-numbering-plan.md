# docs/plans 编号命名整理计划

## 背景

`docs/plans/` 中计划文档数量已达到 61 个，当前文件名只描述主题，无法直接看出生成顺序。用户指出这会影响追溯。该问题成立。

## 目标

1. 给 `docs/plans/` 下所有计划文档增加数字编号前缀。
2. 让目录按文件名排序时基本等于计划生成顺序。
3. 同步更新仓库内引用，避免链接失效。
4. 保留原主题 slug，方便搜索和语义识别。

## 编号规则

采用三位数前缀：

```text
001-project-structure-cleanup-plan.md
002-rotation-detection-plan.md
...
```

排序依据：

1. 优先使用 Git 首次加入该文件的提交时间。
2. 对同一提交中加入、且 Git 日志无法区分先后顺序的早期文件，使用现有 `LastWriteTime` 和文件名作为稳定排序补充。
3. 后续新计划继续使用最大编号加一，不复用旧编号。

## 执行步骤

1. 生成旧文件名到新文件名的映射表。
2. 检查映射表是否覆盖全部计划文件，且无重复编号或重复目标路径。
3. 使用 `git mv` 执行重命名，保留 Git 历史追踪。
4. 全仓替换指向旧计划文件名的引用，包括：
   - `reports/rpd-rotation-detection.md`
   - `docs/README.md`
   - `HANDOFF.md`
   - `docs/decisions/`
   - `docs/research/`
   - `references/`
   - `docs/plans/` 内部交叉引用
5. 生成或更新编号索引，至少在 `docs/README.md` 中保留可读入口。
6. 校验：
   - `docs/plans/` 文件数与执行前一致。
   - 所有计划文件名都匹配 `^\d{3}-.*\.md$`。
   - 仓库中不再引用旧未编号计划路径。
   - `git status` 只包含预期重命名和引用更新。

## 非目标

- 不重写计划正文内容。
- 不改变计划文件所属目录。
- 不删除历史计划。
- 不改变本轮标题栏 crop 审核入口或用户已填写文件。

## 风险与处理

- 风险：引用替换遗漏导致文档链接失效。
  - 处理：用 `rg` 搜索旧文件名残留。
- 风险：同一提交创建的文件缺少精确顺序。
  - 处理：使用稳定排序，并在本计划中说明该顺序是追溯辅助顺序，不声称能完全还原同一提交内的编辑时序。
- 风险：大量改名导致 diff 较大。
  - 处理：先提交本计划回滚点，再单独提交重命名实现。

## 回滚准备

执行重命名前提交本计划、RPD 和 TODO。若引用替换或编号映射出现问题，可回退实现提交，保留本计划重新执行。
