# 标题栏位置多证据仲裁调研笔记

日期：2026-06-28

## 一、学习目标

- 我们真正要解决的问题：验证“先识别标题栏位置，再根据标题栏应在下方或右下方的规则推导旋转角度，旋正后读取图号并重命名单页 PDF”的设计是否合理。
- 本轮不做的事情：不改主流程代码，不训练模型，不调用 VLM，不处理 PDF，不重新跑 OCR，不生成新的人工审核入口。
- 最终沉淀物：资料索引、横向调研笔记、标题栏位置多证据仲裁决策备忘，并标注旧三方投票设计状态。

## 三、对象模型

本项目不应把任务看成“多个模型投票选角度”，而应拆成这些对象：

- `TitleBlockCandidate`：标题栏候选框，包含 OBB/矩形框、所在边侧、面积、长宽比、角度。
- `FrameContactEvidence`：候选是否贴图纸外框线，是否与外框存在明显间隔。
- `TableStructureEvidence`：候选内部是否有标题栏式表格结构，是否区别于均匀明细表或零件线框。
- `FieldClusterEvidence`：OCR 是否命中人员流程字段簇和图纸属性字段簇。
- `OrientationDecision`：由标题栏当前位置映射出的已旋转角度和校正角度。
- `DrawingNumberCandidate`：旋正后从标题栏 OCR 中抽取出的图号候选及命名风险。
- `ArbitrationRecord`：每页保存所有证据、冲突、接受/复核原因和原始输出路径。

## 四、工作协议

推荐工作协议如下：

1. 页面拆分：PDF 拆成一页一个 PDF，这个动作已验证过，不是当前研发风险点。
2. 图像渲染：单页 PDF 渲染为 PNG，用于标题栏定位和证据采集。
3. 候选定位：YOLO/OBB 优先输出标题栏候选框；OpenCV 同时输出图框、表格线、边缘密度和候选几何。
4. 候选真实性确认：OCR 只对候选 crop 跑字段簇，不默认整页 OCR；RapidOCR 当前可作为本地 CPU 证据层。
5. 位置仲裁：根据候选位置、贴边、结构、字段簇、模型分数进行证据仲裁，输出标题栏位置。
6. 旋转校正：按 `rules/mechanical-drawing-rotation.md` 的映射生成校正角度。
7. 图号抽取：只在旋正后标题栏区域抽取图号，并保存 OCR 原文、图号候选和命名决策。
8. 异常分流：证据冲突、候选缺失、字段簇弱、图号重名或命名风险高时才进入 VLM 或人工复核。

## 六、不可变原则

1. 标题栏位置是旋转判断的主对象，角度只是标题栏位置的派生结果。
2. 标题栏检测和图号抽取必须保存中间证据，否则后续无法审计文件重命名是否可靠。
3. 多源证据不是平权多数票，而是按证据类型分工仲裁。
4. 自动化优先，但工业级稳定高于全自动率；只有异常队列才让人参与。
5. 严格数据安全场景下，本地 OpenCV + YOLO/OBB + crop 级 OCR 是默认主线；VLM 是可禁用兜底。
6. 普通机器约束下，不做整页全量 OCR 和默认本地大 VLM 推理，只对标题栏候选 crop 做轻量诊断。

## 七、设计合理性判断

当前设计是合理的，但需要精确定义边界。

合理之处：

- 外部论文已经有“标题栏检测 + 标题栏信息抽取”的图纸处理案例。
- 标准体系将标题栏数据字段视为独立对象，支持保存标题栏 OCR 和字段证据。
- 文档 AI 通用实践支持“先区域定位，再区域解析”。
- OBB、OpenCV、OCR 分别对应位置、结构、文字证据，技术角色清晰。
- 本项目已有 63 张原始样本、20 张 90 度增强样本、YOLO/OBB round2/round3、RapidOCR crop 小实验作为内部实证基础。

需要克制之处：

- 不能说 ISO 公开页直接证明标题栏一定在右下方；具体位置规则仍来自机械制图长期规则和本项目人工确认。
- 不能把建筑图纸论文完全迁移到机械图纸；只能迁移对象模型和处理顺序。
- 不能把 VLM 当默认自动决策者；它适合解释疑难样本，不适合替代本地证据。
- 不能让 OCR 字段簇直接覆盖几何证据；OCR 是真实性和图号抽取证据。

## 八、下一轮验证

下一步最值得做的不是重新讨论投票，而是设计并实现 `ArbitrationRecord`：

- 每页保存标题栏候选框、标题栏位置、旋转角度、校正角度。
- 保存 YOLO/OBB、OpenCV、OCR、VLM 的结构化证据和原始输出路径。
- 保存旋正后标题栏 crop、OCR 原文、字段簇命中、图号候选、最终文件名候选。
- 设置自动接受、VLM 兜底、人工复核三类分流状态。
- 先在既有 hard-case 和 round3 重点样本上回放，不直接批量处理完整 PDF。
